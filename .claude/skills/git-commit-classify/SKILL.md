# git-commit-classify

변경사항을 분석·분류하여 기능 단위로 커밋을 생성하는 스킬

## 실행 절차

### Step 1: 변경사항 전체 확인

아래 명령을 **병렬로** 실행합니다:

```bash
git status
git diff --stat
git diff --cached --stat
```

### Step 2: 변경사항 분류

변경된 파일들을 **기능 단위**로 그룹화합니다.

**분류 기준:**
- 같은 기능/목적에 속하는 파일들을 하나의 그룹으로 묶음
- 서로 다른 기능의 변경은 별도 그룹으로 분리
- 변경 내용을 정확히 파악하기 위해 `git diff` 또는 파일 Read를 통해 실제 코드 변경사항을 확인

**분류 제외 대상:**
- `.DS_Store`, `.env`, `*.log` 등 커밋 불필요 파일

### Step 3: 분류 결과를 사용자에게 확인

분류된 그룹을 아래 형식으로 정리하여 **AskUserQuestion**으로 사용자에게 확인합니다:

```
[그룹 1] {그룹 설명}
  - 파일1
  - 파일2

[그룹 2] {그룹 설명}
  - 파일3
```

- 사용자가 승인하면 Step 4로 진행
- 사용자가 수정을 요청하면 분류를 조정 후 다시 확인

### Step 4: 그룹별 커밋 생성

각 그룹에 대해 **순차적으로** `git-commit` 에이전트(Task tool, subagent_type=git-commit)를 호출합니다.

에이전트에 전달할 정보:
- 해당 그룹에 포함된 파일 목록
- 변경사항 요약 (어떤 기능인지, 무엇이 변경되었는지)
- `git add`할 파일을 명확히 지정

**주의:** 그룹 간 순서대로 커밋합니다. 이전 그룹의 커밋이 완료된 후 다음 그룹을 진행합니다.

### Step 5: 결과 출력

모든 커밋 완료 후 아래 형식으로 결과를 출력합니다:

```
## 커밋 결과

| # | 커밋 메시지 | 파일 수 |
|---|------------|--------|
| 1 | {메시지1} | {n}개 |
| 2 | {메시지2} | {n}개 |

총 {총 커밋 수}개 커밋 생성 완료
```

## 주의사항

- 민감 파일(`.env`, credentials, key 파일 등)은 절대 커밋하지 않습니다.
- 분류가 애매한 경우 사용자에게 확인합니다.
- `git-commit` 에이전트의 커밋 메시지 컨벤션을 따릅니다.
- 변경사항이 하나의 기능에만 해당하면 그룹을 나누지 않고 단일 커밋으로 진행합니다.
